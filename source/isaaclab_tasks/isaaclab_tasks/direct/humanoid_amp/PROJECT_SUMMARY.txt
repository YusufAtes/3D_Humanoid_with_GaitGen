================================================================================
  PROJECT SUMMARY
  3D Extension of Frequency-Based Bipedal Locomotion with Gait Generator Network
================================================================================

Authors:        Yusuf Baran Ates, Omer Morgul
Affiliation:    Dept. Electrical & Electronics Eng., Bilkent University, Ankara, Türkiye
Base Paper:     "Human Imitated Bipedal Locomotion with Frequency Based Gait Generator
                 Network" (arXiv: 2511.17387v1)
Framework:      NVIDIA Isaac Lab (Isaac Sim) with skrl reinforcement learning library
Simulation:     PhysX GPU-accelerated rigid-body simulation at 60 Hz (dt = 1/60 s)
Robot:          28-DoF full 3D humanoid (HUMANOID_28_CFG) with implicit actuators

================================================================================
1. MOTIVATION AND BACKGROUND
================================================================================

The original research paper proposes a lightweight two-stage locomotion framework
for a 7-DoF planar (2D) biped:

  Stage A — Gait Generator Network:
    A fully connected neural network (2-layer MLP, 512 hidden units, ReLU) is
    trained on human motion-capture data from Schreiber & Moissenet (2019). The
    MoCap joint trajectories are transformed into the Fourier (FFT) domain,
    yielding a fixed-size, periodic representation. The network learns the mapping:

        f_θ : (desired_speed, right_leg_length, left_leg_length) → FFT coefficients

    An inverse FFT (IFFT) reconstructs smooth, speed-conditioned joint-angle
    trajectories (hip, knee, ankle for both legs) from these coefficients. This
    frequency-domain encoding naturally enforces periodicity and allows compact
    representation of diverse walking gaits.

  Stage B — PPO Controller:
    A Proximal Policy Optimization (PPO) agent, trained with Generalized Advantage
    Estimation (GAE), uses the gait generator's output as a reference. The agent
    learns torque-level control that tracks these references while satisfying
    balance and energy constraints. Random State Initialization (RSI) and an
    imitation-decay schedule accelerate convergence and prevent reward-hacking.

The paper demonstrates that this controller achieves stable, human-like walking
across speeds of 0–2.2 m/s, ramp angles from −15° to +15°, and terrain
perturbations up to 10 cm — all in a 2D planar simulation (PyBullet).

================================================================================
2. PROJECT GOAL — EXTENDING TO FULL 3D
================================================================================

This project extends the above 2D research to a full 3D humanoid in NVIDIA Isaac
Lab. The 3D extension poses several challenges that the 2D framework does not
encounter:

  • The humanoid has 28 DoF (vs. 7 in 2D), requiring control of arms, torso
    rotation, and full spatial balance.
  • Balance in 3D involves lateral stability and yaw control, not just sagittal
    plane dynamics.
  • The observation space must capture full 3D body state (quaternion orientations,
    3D velocities, key body positions in world frame).
  • The gait generator was trained on sagittal-plane joint angles (hip, knee only),
    so its references cannot directly command all 28 DoF.

The project was developed in three progressive stages, each building on the
lessons and limitations of the previous one.

================================================================================
3. PROJECT ARCHITECTURE — THREE STAGES
================================================================================

-------------------------------------------------------------------------------
3.1  Stage 1: Direct 3D Imitation (locomotion_env.py)
-------------------------------------------------------------------------------

Location:  source/isaaclab_tasks/isaaclab_tasks/direct/locomotion/

This was the initial, straightforward attempt to replicate the 2D paper's
approach directly in 3D. The controller uses:

  Environment:     LocomotionEnv (DirectRLEnv subclass)
  Control Mode:    Torque control (set_joint_effort_target)
  Gait Generator:  SimpleFCNN outputting 136 FFT coefficients (no period output)
                   → IFFT → 1920-step × 4-joint reference trajectory
  Observations:    Torso height, local velocity, angular velocity, yaw, roll,
                   angle-to-target, up/heading projections, scaled DOF positions,
                   DOF velocities, actions, plus reference joint angles at
                   current time (t), t+1, and t+10.
  Rewards:         Progress toward target, alive bonus, upright posture, heading
                   alignment, imitation (hip/knee L2 distance to reference),
                   velocity tracking, minus action/energy penalties.
  Reset:           Random State Initialization (RSI) — the humanoid is initialized
                   at a random phase of the reference gait cycle, with hip/knee
                   angles set from the gait generator and root height adjusted
                   to compensate for leg shortening.

  Key Observation:
    The full reference trajectory (current + future frames) is appended to the
    observation vector, giving the policy explicit lookahead into the gait plan.

  OUTCOME:
    This direct imitation approach proved INSUFFICIENT for robust 3D locomotion.
    The humanoid struggled to maintain lateral balance and natural motion quality.
    Passing raw FFT-reconstructed trajectories as observations led to a very high-
    dimensional observation space and did not transfer well from the sagittal-plane
    training data to full 3D control.

-------------------------------------------------------------------------------
3.2  Stage 2: AMP + Phase-Based Gait Generator (humanoid_amp_im_env.py)
-------------------------------------------------------------------------------

Location:  source/isaaclab_tasks/isaaclab_tasks/direct/humanoid_amp/

To overcome the limitations of direct imitation, a fundamentally different
architecture was adopted with two key changes:

  CHANGE 1 — Phase-Based Reference (instead of full FFT trajectories):
    Instead of outputting the entire FFT coefficient vector and reconstructing the
    full trajectory in the observation, the gait generator network was modified to
    output 137 values: 136 FFT coefficients + 1 period value. The IFFT-
    reconstructed reference is stored internally, and only the current PHASE
    (a single scalar in [0, 1]) and the desired speed are appended to the
    observation. This dramatically reduces the observation dimension while still
    providing the policy with the cyclic timing information it needs.

    The phase is computed as:
        phase_step = (episode_step + random_start_idx) mod period
        phase = phase_step / period

    The actual joint reference values (hip, knee angles) are retrieved from the
    stored reference using the phase_step index and used exclusively in the reward
    computation — not in the observation. This prevents observation bloat and
    forces the policy to learn an internal representation of the gait pattern.

  CHANGE 2 — Adversarial Motion Priors (AMP) Discriminator:
    An AMP-style discriminator network is integrated alongside the PPO policy.
    The discriminator is trained to distinguish between the humanoid's actual
    motion and reference motions loaded from motion-capture data (humanoid_walk.npz).
    This provides a "style reward" that encourages natural, human-like whole-body
    motion — complementing the explicit imitation reward that only covers hip and
    knee joints.

    The AMP observation (81-D) includes:
      • 28 joint positions
      • 28 joint velocities
      • 1 root body height
      • 6 root orientation (tangent + normal vectors from quaternion)
      • 3 root linear velocity
      • 3 root angular velocity
      • 12 key body relative positions (right_hand, left_hand, right_foot,
        left_foot relative to torso)

  CHANGE 3 — Position Control:
    The control mode was switched from torque control to position control
    (set_joint_position_target). Actions are mapped to target joint positions via:
        target = action_offset + action_scale × action
    where action_offset and action_scale are computed from the joint soft limits.

  Environment:     HumanoidAmpEnv (DirectRLEnv subclass)
  Control Mode:    Position control (set_joint_position_target)
  Gait Generator:  SimpleFCNN outputting 137 values (136 FFT + 1 period)
                   → IFFT → 96-step × 4-joint reference (looped via modular phase)
  Observations:    81-D AMP observation + phase (1) + desired_speed (1) = 83-D total
  AMP Obs:         81-D (fed to discriminator, first 81 of policy obs)
  Action Space:    28-D (joint position targets for all DoF)

  Reward Structure:
    The total reward is a combination of task reward and AMP style reward:

    Task Reward (computed in _get_rewards):
      • Imitation reward (hip tracking):
          w_hip × exp(−5.0 × ‖joint_hip − ref_hip‖₂)
      • Imitation reward (knee tracking):
          w_knee × exp(−5.0 × ‖joint_knee − ref_knee‖₂)
      • Forward velocity tracking:
          w_fwd × exp(−3.0 × |v_forward − v_desired|)
      • Lateral velocity penalty:
          w_lat × exp(−3.0 × |v_lateral|)
      • Yaw velocity penalty:
          w_yaw × exp(−3.0 × |ω_yaw|)
      • Death penalty:
          −1.75 if torso height < termination_height

    Speed-dependent imitation decay:
      For desired_speed > 0.5, the imitation weights are scaled by
      exp(−1.5 × (speed − 0.5)), while velocity weights are inversely scaled.
      This allows the policy to prioritize velocity tracking at higher speeds
      where the gait generator references become less reliable.

    AMP Style Reward (from discriminator):
      The discriminator score is transformed into a reward signal weighted by
      style_reward_weight (0.6). The final reward combines:
          total = task_reward_weight × task_reward + style_reward_weight × style_reward

  Training Configuration (skrl_walk_amp_im_cfg.yaml):
    Agent:            AMP (Adversarial Motion Priors with PPO backbone)
    Policy Network:   GaussianMixin, [1024, 512] layers, ReLU activation
    Value Network:    DeterministicMixin, [1024, 512] layers, ReLU
    Discriminator:    DeterministicMixin, [1024, 512] layers, ReLU
    Learning Rate:    5.0e-05
    Discount:         0.99, GAE λ = 0.95
    Rollout Length:    16 steps
    Mini-batches:     2
    Learning Epochs:  4
    Entropy Coeff:    3.0e-04
    Task Reward Wt:   0.8
    Style Reward Wt:  0.6
    Disc. Batch Size: 4096
    Disc. Reward Scale: 2.0
    Total Timesteps:  1,000,000
    Motion Dataset:   200,000 samples (RandomMemory)
    Reply Buffer:     1,000,000 samples (prevents discriminator overfitting)

  Registered Gym ID: Isaac-Humanoid-AMP-Imp-Direct-v0

  Reset / RSI:
    At each environment reset, a random speed is sampled from U(0.1, 2.4) m/s
    (normalized to [0, 1] by dividing by 2.4). The gait generator produces a
    reference trajectory for that speed. A random starting index within the gait
    cycle is chosen, and the humanoid's hip and knee joints are initialized to
    match the reference at that phase. Root height is adjusted upward (+0.4) to
    prevent ground penetration.

  Demo / Evaluation Modes:
    The environment supports three demo modes for evaluation:
      • "vel"   — Fixed velocity testing (set via set_test_speed)
      • "ramp"  — Inclined ground plane (slope angle from test_slope_deg)
      • "noise" — Rough terrain with configurable amplitude, seed, and type
                   (random uniform or wave heightfields)

-------------------------------------------------------------------------------
3.3  Stage 3: Waypoint Path Following (humanoid_amp_im_path_env.py)
-------------------------------------------------------------------------------

Location:  source/isaaclab_tasks/isaaclab_tasks/direct/humanoid_amp/

This stage extends the walking controller to follow paths defined by sequential
waypoint targets. The humanoid must not only walk at desired speeds with natural
gait, but also navigate toward dynamically generated waypoints in the XY plane.

  Environment:     HumanoidAmpEnv (same class name, different module)
  Control Mode:    Position control (same as Stage 2)
  Observations:    81-D AMP obs + phase (1) + desired_speed (1) +
                   target_distance (1) + heading_error (1) = 85-D total

  Additional State Variables:
    • target_pos [N, 2]:        Current waypoint XY position
    • past_torso_pos [N, 2]:    Previous torso XY position (for progress reward)
    • current_target_dist [N]:  Distance to current waypoint
    • heading_error [N]:        Angular error between robot heading and target

  Waypoint Generation (_resample_waypoint_targets):
    New waypoints are placed 1.0 m ahead of the robot's current position, within
    ±20° of the robot's current yaw heading. The yaw is extracted from the torso
    quaternion (w,x,y,z → atan2 decomposition). When a waypoint is reached
    (distance < 0.1 m), a new one is immediately sampled.

  Reward Structure (two modes):

    Mode A — Initial Training (second_training = False):
      Same as Stage 2: imitation + velocity rewards with speed-dependent
      imitation decay. No path-following rewards are applied. This lets the
      policy first learn stable walking before adding navigation objectives.

    Mode B — Second Training / Curriculum (second_training = True):
      Reduced imitation weights (0.125 for hip and knee) and modified velocity
      weights (0.25 forward, 0.1 lateral, 0.1 yaw), plus three new rewards:

      • Target Distance Reward (w = 0.5):
          Measures velocity toward the target:
            vel_to_target = (prev_dist − curr_dist) / dt
          Reward = 0.5 × exp(−4.0 × |vel_to_target − desired_speed|)

      • Target Heading Reward (w = 0.25):
          heading_error = atan2(sin(θ_target − θ_robot), cos(θ_target − θ_robot))
          Reward = 0.25 × exp(−4.0 × |heading_error|)

      • Target Reached Bonus (w = 1.0):
          +1.0 reward when distance < 0.1 m, then immediately resample new target

      This implements a form of curriculum learning: the policy is first trained
      to walk naturally (Mode A), and then fine-tuned with path-following
      objectives (Mode B) using a reduced style reward weight.

  Training Configuration (skrl_walk_amp_im_path_cfg.yaml):
    Same AMP agent structure as Stage 2, with modifications:
    Learning Rate:     1.0e-04 (2× higher than walking-only)
    Entropy Coeff:     1.0e-04 (reduced for more deterministic paths)
    Task Reward Wt:    1.0 (increased from 0.8)
    Style Reward Wt:   0.75 (increased from 0.6; reduced to 0.25 for second training)
    Total Timesteps:   1,000,000

  Registered Gym ID: Isaac-Humanoid-AMP-Imp-Path-Direct-v0

================================================================================
4. GAIT GENERATOR NETWORK — DETAILED DESCRIPTION
================================================================================

Architecture:    SimpleFCNN (fully connected neural network)
                 3 hidden layers × 512 units, LeakyReLU(0.1) activation
Input (3-D):     [desired_speed / 2.4,  right_leg_length / 1.0,  left_leg_length / 1.0]
                 (speed normalized to ~[0,1], leg lengths normalized)
Output (137-D):  [136 FFT coefficients,  1 period value]

Post-Processing Pipeline:
  1. Extract first 136 values → denormalize using pre-computed mean/std
  2. Reshape: (136,) → (17 freqs, 4 joints, 2 real/imag) → transpose to (4, 2, 17)
  3. Combine real + imaginary → complex array (4 joints × 17 frequencies)
  4. Inverse FFT (n=32): reconstruct 32-point time-domain signal per joint
  5. If simulation dt < 0.1s: resample from 10 Hz to (1/dt) Hz using Fourier method
  6. Result: [T, 4] tensor of joint angles (right_hip, right_knee, left_hip, left_knee)
  7. Extract period value (output[136]) → scale by episode_len → clamp ≥ 12

The 4 output joints correspond to sagittal-plane hip and knee angles for both
legs. These are used for imitation reward computation but NOT for directly
commanding the 28-DoF humanoid.

Trained Model:   FINAL_BEST_MODEL.pth (2.3 MB)
Normalization:   mean.npy, std.npy (pre-computed from training data)
Training Data:   Human gait dataset by Schreiber & Moissenet (2019)
                 1143 gait trials, 50 participants, 5 walking regimes
                 Markers mapped to hip/knee/ankle angles, FFT-encoded

  • Inclined ramp: ground plane rotated around Y-axis by test_slope_deg
  • Noisy terrain: procedurally generated heightfield via TerrainImporter

    - Wave terrain: configurable amplitude and number of waves
    - Seed-controlled for reproducibility across evaluation trials

9. KEY DESIGN DECISIONS AND INSIGHTS
================================================================================

  1. Phase-Only Observation:
     Adding only the phase scalar (instead of full reference trajectories) to the
     observation keeps the input compact and forces the policy to learn an internal
     model of the gait cycle. The reference joint angles are used only in the
     reward, acting as a soft constraint rather than an explicit tracking target.

  2. AMP for 3D Naturalness:
     The gait generator only produces sagittal-plane hip/knee references — it
     cannot guide arm swing, torso twist, or foot placement in 3D. The AMP
     discriminator fills this gap by learning a holistic "naturalness" score from
     full-body MoCap data, encouraging coordinated whole-body motion.

  3. Speed-Dependent Imitation Decay:
     At higher speeds, the gait generator references become less reliable (they
     were trained on walking data up to ~2.4 m/s). The exponential decay of
     imitation weights at higher speeds allows the policy to deviate from
     potentially inaccurate references and prioritize velocity tracking.

  4. Curriculum for Path Following:
     Path following is introduced as a second training phase. The policy first
     learns stable, natural walking (Mode A with full imitation + AMP), then is
     fine-tuned with navigation rewards (Mode B with reduced imitation, added
     distance/heading/reached rewards). This prevents the policy from learning
     unnatural shortcuts to reach waypoints.

  5. Position vs. Torque Control:
     Switching from torque control (Stage 1) to position control (Stages 2-3)
     significantly improved stability. Position control provides implicit
     compliance through the actuator model, making it easier for the policy to
     maintain balance in 3D.

  6. RSI from Gait Generator:
     Random State Initialization uses the gait generator's own output to place
     the humanoid at diverse starting poses within the gait cycle. This prevents
     the policy from overfitting to a single initial configuration and improves
     generalization across the full gait phase.

10. TRAINING AND EVALUATION WORKFLOW
================================================================================

  Training:
    1. Configure environment (num_envs=4096 for training, demo_mode=False)
    2. Run skrl AMP training for 1M timesteps
    3. For path following: first train with second_training=False, then fine-tune
       with second_training=True and reduced style_reward_weight

  Evaluation / Demo:
    1. Set demo_mode=True, num_envs=1
    2. Choose demo_type: "vel", "ramp", or "noise"
    3. Set test_speed via set_test_speed() method
    4. For ramp: set test_slope_deg in config
    5. For noise: set noise_amplitude, noise_seed, noise_type in config
    6. Run play.py script to load checkpoint and evaluate

  Metrics:
    • Average forward speed (computed after initial 30-step warmup)
    • Episode survival time
    • Max traversal range on noisy/ramped terrain
    • Velocity tracking error
    • Waypoint reach rate (path following)
